# ETL Process Documentation — BI Sales Project

## 1. Overview

This document describes the Extract, Transform, Load (ETL) process used to convert the raw Sample Superstore flat file into the star schema defined in `data-model.md`. The process was implemented in Python using `pandas`, inside a Jupyter notebook (`explore.ipynb`) in VS Code.

At a high level, the ETL followed this order:
1. **Extract** — load the raw source file.
2. **Transform** — derive each dimension table, then build the fact table by merging surrogate keys back onto the source data.
3. **Load** — export all resulting tables as CSV files, ready to be loaded into a reporting tool (Power BI).

---

## 2. Extract

The source file (`Sample_Superstore.csv.xlsx`) was read using `pandas.read_excel()`. Despite the `.csv` text in its filename, the actual file format was Excel, confirmed by inspecting the file extension directly rather than assuming from the name.

No cleaning was required at this stage, since prior data profiling confirmed no missing values, no duplicate rows, and no anomalous numeric values in the source data.

---

## 3. Transform

### 3.1 Dimension tables built first

Each dimension table was derived by selecting the relevant descriptive columns from the source data and removing duplicates, so that each dimension row represents one unique entity:

- **Dim_Customer** — Customer ID, Segment
- **Dim_Product** — Product ID, Product Name, Category, Sub-Category
- **Dim_Location** — Country, State, City, Region (assigned a generated surrogate key, `Location ID`, since no single natural key exists for this combination)
- **Dim_Shipping** — Ship Mode (assigned a generated surrogate key, `Ship Mode ID`)

### 3.2 Dim_Date — built independently

Unlike the other dimensions, Dim_Date was not extracted directly from a single column. Instead, all unique dates from both `Order Date` and `Ship Date` were combined into one master list, since the date dimension needs to support both roles. Standard calendar attributes were then derived: Year, Quarter, Month, Month Name, Day, Weekday, and an `Is Weekend` boolean flag.

**Design rationale:** generating the date dimension independently, rather than pulling it from one column, is standard BI practice. It ensures the dimension covers every date the fact table will ever need to reference, and makes rich calendar attributes available for reporting without recalculating them in every query.

### 3.3 Fact_Sales — merging surrogate keys

The fact table was built by taking the original flat file and merging in the surrogate keys generated by each dimension table:

- `Order Date` was matched against `Dim_Date` to retrieve `Order Date ID`.
- `Ship Date` was matched against the same `Dim_Date` table a second time to retrieve `Ship Date ID`. This is the practical implementation of the role-playing dimension described in the data model: one physical dimension table serving two distinct analytical roles within the fact table.
- `Country`, `State`, `City`, and `Region` were matched together against `Dim_Location` to retrieve `Location ID`.
- `Ship Mode` was matched against `Dim_Shipping` to retrieve `Ship Mode ID`.

All merges used `how='left'`, preserving every row of the original fact data regardless of match outcome — this makes it possible to detect any failed matches afterward rather than silently dropping rows.

### 3.4 Derived measure: Shipping Duration

A new measure, not present in the source data, was calculated during transformation:

```
Shipping Duration (days) = Ship Date − Order Date
```

This was added directly to Fact_Sales. It supports analysis that connects the Date and Shipping dimensions indirectly through the fact table (e.g. average shipping duration per Ship Mode) — dimensions are never linked to each other directly in a star schema, so this measure is what makes that kind of cross-dimension analysis possible.

---

## 4. Verification

After constructing Fact_Sales, two checks were run to confirm the transformation was correct:

1. **Row count check** — confirmed Fact_Sales contained exactly 9,994 rows, matching the original source file. This ruled out row duplication, which can occur if a merge key in a dimension table isn't fully unique.
2. **Null check** — confirmed zero missing values across all foreign key columns (`Order Date ID`, `Ship Date ID`, `Location ID`, `Ship Mode ID`). A null in any of these would indicate a merge failed to find a matching dimension row, for example due to inconsistent text formatting between the fact and dimension values.

Both checks passed cleanly, confirming the ETL transformation preserved data integrity with no row loss, duplication, or broken relationships.

---

## 5. Load

All six resulting tables were exported to CSV using `to_csv()`:

- `Fact_Sales.csv`
- `Dim_Date.csv`
- `Dim_Customer.csv`
- `Dim_Product.csv`
- `Dim_Location.csv`
- `Dim_Shipping.csv`

These files form the complete star schema on disk and are the data source for the Power BI dashboard built in the next project stage.
